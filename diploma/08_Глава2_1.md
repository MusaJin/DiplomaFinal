# ГЛАВА 2. ПРОЕКТНАЯ ЧАСТЬ

## 2.1 Информационное обеспечение задачи

Информационное обеспечение информационной системы охватывает совокупность решений, связанных с организацией хранения, структурирования и представления данных, циркулирующих в системе. Качество информационного обеспечения непосредственно определяет корректность работы всех функциональных модулей: ошибки в проектировании схемы данных приводят к аномалиям при обновлении, избыточному дублированию информации и снижению производительности запросов. Вместе с тем недостаточно нормализованная схема, напротив, не позволяет эффективно выражать сложные взаимосвязи между сущностями. Ниже последовательно рассматриваются информационная модель системы, применяемые классификаторы, характеристика входных данных и базы данных, а также описание результатной информации.

Информационная модель системы отражает состав сущностей предметной области, их атрибуты и взаимосвязи. Центральными сущностями модели являются пользователь (User), новость (News) и образовательный ресурс (Resource). Каждый пользователь характеризуется уникальным идентификатором, полным именем, адресом электронной почты, хешированным паролем и ролью. Роль является перечислимым атрибутом с тремя допустимыми значениями: STUDENT, TEACHER, ADMIN. Адрес электронной почты является уникальным идентификатором пользователя при аутентификации и выступает альтернативным ключом таблицы. Сущность пользователя связана с сущностями новостей и ресурсов отношением «один ко многим» (один пользователь может быть автором многих публикаций), 
Сущность новости включает следующие атрибуты: уникальный числовой идентификатор, заголовок (строка, не более двухсот пятидесяти символов), краткое описание (строка, не более пятисот символов), полный текст (текстовое поле без ограничения длины), URL изображения (необязательный атрибут), признак опубликованности (булев атрибут, по умолчанию false), дата публикации (заполняется автоматически при установке признака опубликованности), дата создания и дата последнего обновления записи. Новость связана с категорией необязательным внешним ключом: при отсутствии категории новость отображается как некатегоризированная. Связь с автором является обязательной — каждая новость имеет ровно одного автора из числа пользователей с ролью ADMIN.

Сущность образовательного ресурса обладает схожей структурой, но дополнена атрибутами, специфичными для учебных материалов. Тип ресурса является перечислимым атрибутом со значениями LINK (гиперссылка на внешний ресурс), FILE (загружаемый файл, ссылка на который хранится в системе) и TEXT (текстовое содержимое, хранящееся непосредственно в базе данных). В зависимости от типа заполняется либо атрибут fileUrl (URL ресурса или загруженного файла), либо атрибут textContent (текст, представленный непосредственно в записи). Ресурс также связан с категорией необязательной связью и имеет обязательную связь с автором — пользователем с ролью TEACHER или ADMIN.

Сущность категории включает числовой идентификатор, наименование и тип. Тип категории определяет область её применения: значение NEWS означает, что категория используется для классификации новостей, RESOURCE — для образовательных ресурсов, COMMON — для обеих сущностей одновременно. Такое разграничение позволяет исключить ситуацию, при которой категория, созданная для организации новостей, отображается в фильтрах каталога образовательных ресурсов, где её присутствие было бы семантически некорректным. Категория связана с новостями и ресурсами отношением «один ко многим»: одна категория может быть присвоена множеству публикаций.

Применяемые в системе классификаторы и системы кодирования обеспечивают единообразное представление перечислимых атрибутов во всех слоях приложения. Кодирование ролей пользователей основано на трёх константных значениях: ADMIN, TEACHER, STUDENT. Эти значения хранятся в базе данных как тип перечисления PostgreSQL и транслируются в TypeScript enum на уровне Prisma-сгенерированных типов, что обеспечивает строгий контроль допустимых значений на всём протяжении стека приложения. Кодирование типов образовательных ресурсов аналогично реализовано через enum: LINK, FILE, TEXT. Кодирование типов категорий: NEWS, RESOURCE, COMMON. Использование enum вместо строковых констант исключает орфографические ошибки при формировании запросов и делает код самодокументируемым.

Характеристика первичных документов с нормативно-справочной и входной оперативной информацией описывает источники данных, поступающих в систему. Нормативно-справочной информацией системы являются классификаторы категорий, создаваемые администратором в момент первоначальной настройки системы и пополняемые по мере необходимости. Входной оперативной информацией являются новостные публикации и записи об образовательных ресурсах, создаваемые пользователями с соответствующими правами в процессе текущей деятельности. Каждая новость при создании требует заполнения следующих обязательных полей: заголовок, краткое описание, полный текст. Поля URL изображения и категории являются необязательными. Запись об образовательном ресурсе требует обязательного указания наименования, описания и типа, а также соответствующего типу обязательного атрибута — URL или текстового содержимого.

Характеристика базы данных включает описание всех таблиц с указанием их полей, типов данных и ограничений целостности. Таблица users содержит поля: id (integer, primary key, autoincrement), fullName (varchar(255), not null), email (varchar(255), unique, not null), password (varchar(255), not null — хеш bcrypt), role (enum Role, not null, default STUDENT), createdAt и updatedAt (timestamp). Таблица categories: id, name (varchar(100), not null), type (enum CategoryType, not null), временны́е метки. Таблица news: id, title (varchar(250), not null), shortDescription (varchar(500), not null), content (text, not null), imageUrl (varchar(500), nullable), published (boolean, default false), publishedAt (timestamp, nullable), authorId (foreign key → users.id, onDelete RESTRICT), categoryId (foreign key → categories.id, onDelete SET NULL), временны́е метки. Таблица resources: id, title (varchar(250), not null), description (varchar(500), not null), type (enum ResourceType, not null), fileUrl (varchar(500), nullable), textContent (text, nullable), authorId (foreign key → users.id, onDelete RESTRICT), categoryId (foreign key → categories.id, onDelete SET NULL), временны́е метки. 
Ограничения ссылочной целостности определены следующим образом. Внешние ключи authorId в таблицах news и resources имеют политику ON DELETE RESTRICT, что запрещает удаление пользователя, являющегося автором хотя бы одной публикации. Внешние ключи categoryId имеют политику ON DELETE SET NULL: при удалении категории соответствующие публикации не удаляются, а теряют категориальную принадлежность. 
Характеристика результатной информации описывает данные, формируемые системой в ответ на запросы мобильного клиента. Основными результирующими структурами являются: список новостей с вложенными объектами автора и категории; детальное представление новости, дополняющее список полем content; список образовательных ресурсов; детальное представление ресурса с добавлением поля textContent. Все списковые представления возвращаются в формате JSON-массива и упорядочены по убыванию даты создания. Ответы на запросы авторизации содержат объект с полями accessToken, refreshToken и объект user. Все ответы с ошибками содержат поле message с описанием ошибки и соответствующий HTTP-статус-код.
