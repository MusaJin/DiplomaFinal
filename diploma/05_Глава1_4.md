## 1.4 Постановка задачи

Постановка задачи является центральным методологическим этапом проектирования информационной системы, на котором абстрактная потребность в автоматизации трансформируется в конкретную и верифицируемую систему требований. Качество этого перехода во многом определяет как техническую корректность реализации, так и практическую применимость готового продукта: система, разработанная без чёткой постановки задачи, рискует оказаться либо избыточной по функциональности, либо неспособной закрыть ключевые потребности целевой аудитории. В рамках настоящей работы постановка задачи строится на результатах аналитического этапа — описании предметной области, характеристике информационных потоков и экономическом обосновании необходимости автоматизации — и формализует их в виде функциональных и нефункциональных требований к разрабатываемой системе.

Целью и назначением разрабатываемой автоматизированной системы является обеспечение централизованного, управляемого и надёжного информационного взаимодействия между участниками образовательного процесса университета посредством мобильного приложения. Система предназначена для автоматизации следующих ключевых процессов: создания и публикации институциональных новостей с обеспечением их оперативной доступности для всех зарегистрированных пользователей; структурированного хранения и предоставления доступа к образовательным ресурсам различных типов; управления учётными записями пользователей с разграничением прав по ролевому принципу; классификации контента через систему настраиваемых категорий. Область применения системы ограничена рамками одного высшего учебного заведения и не предполагает интеграции с внешними образовательными платформами в рамках текущей версии продукта.

Функциональные требования определяют, что именно система обязана уметь делать, и задают критерии приёмки реализованного решения. Первой группой функциональных требований является управление пользователями: система должна обеспечивать регистрацию новых учётных записей с указанием полного имени, адреса электронной почты и пароля; аутентификацию пользователей с выдачей и обновлением JWT-токенов; назначение и хранение роли пользователя (STUDENT, TEACHER, ADMIN). Безопасность аутентификационного механизма обеспечивается хешированием паролей с применением алгоритма bcrypt и ограниченным сроком действия токенов доступа.

Второй группой функциональных требований является управление новостями. Пользователи с ролью ADMIN должны иметь возможность создавать новостные публикации с заполнением обязательных полей (заголовок, краткое описание, полный текст) и опциональных атрибутов (изображение, категория); редактировать содержимое ранее опубликованных новостей; удалять новостные записи. Все зарегистрированные пользователи независимо от роли должны иметь возможность просматривать список опубликованных новостей с фильтрацией по категориям и открывать детальное представление отдельной публикации. Список новостей должен отображаться в порядке убывания даты публикации.

Третья группа функциональных требований охватывает управление образовательными ресурсами. Пользователи с ролью TEACHER и ADMIN должны иметь возможность создавать записи об образовательных ресурсах с указанием наименования, описания, типа ресурса (LINK, FILE, TEXT) и соответствующих атрибутов (URL, файл или текстовое содержимое); редактировать и удалять созданные ресурсы. Все зарегистрированные пользователи должны иметь возможность просматривать каталог образовательных ресурсов с фильтрацией по категориям и типам, а также открывать детальное представление отдельного ресурса с доступом к его содержимому. Такая модель разграничения отражает реальную практику университета: преподаватели являются основными поставщиками учебных материалов, тогда как студенты выступают в роли их потребителей.

Четвёртой группой функциональных требований является управление категориями. Пользователи с ролью ADMIN должны иметь возможность создавать категории с указанием наименования и области применения (NEWS, RESOURCE или COMMON); редактировать и удалять существующие категории. Категории используются для классификации как новостей, так и образовательных ресурсов, обеспечивая единую таксономическую основу системы. Набор категорий не является фиксированным и может пополняться по мере развития информационной среды учреждения.

Нефункциональные требования задают качественные характеристики системы, не связанные непосредственно с её функциональным содержанием, но в равной мере определяющие практическую ценность разработки. В части производительности система должна обеспечивать время отклика на запросы к API не более пятисот миллисекунд при нормальной нагрузке; список новостей должен загружаться и отображаться на экране мобильного устройства в течение двух секунд от момента открытия соответствующего экрана приложения. Эти показатели соответствуют ожиданиям пользователей мобильных приложений, для которых задержка более двух-трёх секунд является значимым негативным фактором восприятия.

Требования безопасности предусматривают хранение паролей исключительно в виде криптографических хешей (bcrypt, стоимостный фактор не менее десяти); передачу всех данных между клиентом и сервером по защищённому каналу (HTTPS в продуктивном окружении); использование JWT-токенов с ограниченным сроком действия (access token — двадцать четыре часа, refresh token — семь суток); проверку роли пользователя на серверной стороне при каждом запросе, требующем привилегированного доступа. Хранение токенов аутентификации на клиентской стороне осуществляется в защищённом хранилище Expo SecureStore, что исключает их доступность через стандартные механизмы межприложенческого обмена данными.

Требование кроссплатформенности предполагает корректную работу мобильного клиента как на устройствах под управлением Android (версии 8.0 и выше), так и на устройствах под управлением iOS (версии 13 и выше). Выбор React Native в качестве фреймворка разработки обеспечивает выполнение этого требования за счёт единой кодовой базы, транслируемой в нативные компоненты каждой целевой платформы. Разработка двух отдельных приложений на Swift/Kotlin в данном случае нецелесообразна как с точки зрения трудозатрат, так и с точки зрения долгосрочного сопровождения.

Требование масштабируемости системы определяет, что архитектурные решения, принятые при проектировании серверной части, должны обеспечивать возможность горизонтального масштабирования — увеличения числа экземпляров серверного приложения за счёт балансировки нагрузки — без внесения изменений в логику приложения. Отсутствие состояния (stateless) на уровне REST API и хранение сессионных данных в JWT-токенах, а не в серверной памяти, являются ключевыми архитектурными решениями, обеспечивающими это свойство. При возрастании числа пользователей сверх расчётных показателей системный администратор должен иметь возможность масштабировать систему путём запуска дополнительных экземпляров серверного приложения без изменения его кода.

Общая характеристика организации решения задачи предполагает описание взаимодействия основных компонентов системы. Серверная часть на Node.js с Express обеспечивает обработку HTTP-запросов от мобильного клиента, выполняет бизнес-логику и взаимодействует с базой данных через Prisma ORM. Мобильный клиент на React Native с Expo взаимодействует с сервером через REST API, сохраняет аутентификационные токены в SecureStore и отображает полученный контент в соответствии с назначенной пользователю ролью. Связь между клиентом и сервером осуществляется через HTTP/HTTPS с использованием JSON как формата обмена данными. Такая клиент-серверная архитектура является стандартной для мобильных приложений и обеспечивает чёткое разделение ответственности между компонентами системы.
