## 2.5 Описание контрольного примера реализации проекта

Контрольный пример реализации служит практическим подтверждением работоспособности разработанной системы и демонстрирует корректность функционирования всех ключевых модулей в условиях, приближённых к реальной эксплуатации. В рамках контрольного примера рассматриваются три основных сценария, охватывающих деятельность каждой из трёх ролей пользователей: администратора, преподавателя и студента. Последовательность сценариев выстроена таким образом, что действия пользователей с привилегированными ролями порождают контент, с которым впоследствии взаимодействуют пользователи с базовыми правами, что позволяет проверить сквозное взаимодействие всех компонентов системы.

Для проведения контрольного примера в базе данных предварительно созданы тестовые учётные записи трёх пользователей: администратор с адресом электронной почты admin@university.ru и полным именем «Администратор Системы», преподаватель с адресом teacher@university.ru и именем «Иванов Иван Иванович», а также студент с адресом student@university.ru и именем «Петров Пётр Петрович». Также созданы тестовые категории: «Административные объявления» (тип NEWS) и «Учебные материалы» (тип RESOURCE). Серверная часть запущена локально на порту 3001; мобильный клиент функционирует в Android-эмуляторе, взаимодействующем с сервером через адрес 10.0.2.2 с проброской порта через adb reverse.

Сценарий первый: администратор создаёт и публикует новость. Администратор запускает мобильное приложение и на экране входа, представленном на рисунке 1, вводит учётные данные admin@university.ru и соответствующий пароль. После нажатия кнопки «Войти» клиент выполняет POST-запрос к /api/auth/login; сервер верифицирует учётные данные, генерирует пару токенов и возвращает их вместе с объектом пользователя. Клиент сохраняет токены в SecureStore, устанавливает данные пользователя в Zustand store и переходит на главный экран приложения, показанный на рисунке 2. На главном экране администратор видит агрегированную ленту последних новостей и ресурсов; в нижней навигационной панели помимо стандартных вкладок отображается вкладка «Управление», доступная только пользователям с ролью ADMIN.

Администратор переходит на экран управления новостями, который отображает текущий список всех публикаций — включая черновики — с индикаторами статуса, что представлено на рисунке 7. Для создания новой новости нажимается кнопка «Создать» в правом верхнем углу экрана, после чего открывается форма, показанная на рисунке 8.

Рисунок 12 — Форма создания и редактирования новости

В форму вводятся следующие данные: «Заголовок» — «Изменение расписания на период сессии», «Краткое описание» — «В связи с началом экзаменационной сессии вносятся изменения в расписание занятий», «Полный текст» — развёрнутый текст объявления; из выпадающего списка выбирается категория «Административные объявления». После нажатия кнопки «Сохранить» клиент формирует POST-запрос к /api/news с телом, содержащим все введённые данные. Сервер принимает запрос, проверяет токен администратора, создаёт запись в таблице news с признаком published=true и текущей датой в поле publishedAt, после чего возвращает клиенту объект созданной новости со статусом 201. Экран административного списка обновляется: новая запись отображается первой с меткой «Опубликовано». Таким образом, первый сценарий подтверждает корректность взаимодействия модулей аутентификации и создания контента.

Сценарий второй: преподаватель создаёт образовательный ресурс. Преподаватель авторизуется в приложении с учётными данными teacher@university.ru через экран входа, представленный на рисунке 1, после чего клиент определяет роль TEACHER и формирует соответствующий набор элементов интерфейса. В разделе «Ресурсы», каталог которого показан на рисунке 5, в правом верхнем углу присутствует кнопка «Добавить», недоступная пользователям с ролью STUDENT. Преподаватель нажимает эту кнопку и переходит на форму создания ресурса, представленную на рисунке 10. В поле «Название» вводится «Лекция №3: Основы реляционных баз данных», в поле «Описание» — «Конспект лекции по теме нормализации отношений, первая, вторая и третья нормальные формы», тип ресурса устанавливается в значение LINK, в поле URL вводится ссылка на внешний образовательный ресурс, выбирается категория «Учебные материалы». После нажатия кнопки «Сохранить» клиент отправляет POST-запрос к /api/resources; сервер проверяет роль пользователя (TEACHER допустима для создания ресурсов), создаёт запись с authorId, соответствующим идентификатору текущего пользователя, и возвращает созданный объект. Преподаватель возвращается к каталогу ресурсов, где созданная запись отображается первой с иконкой типа LINK синего цвета; рядом с карточкой доступно меню редактирования, скрытое от пользователей с ролью STUDENT.

Сценарий третий: студент просматривает новости и ресурсы. Студент авторизуется с учётными данными student@university.ru и переходит в раздел «Новости». Опубликованная администратором новость «Изменение расписания на период сессии» отображается первой в ленте. Студент открывает её детальную страницу, показанную на рисунке 4, которая отображает изображение новости, категориальный бейдж «Административные объявления», заголовок, метаданные автора и дату публикации, краткое описание в выделенном блоке с левой синей полосой и полный текст публикации. Студент также переходит в раздел «Ресурсы» и находит созданный преподавателем материал; фильтрация по категории «Учебные материалы» позволяет быстро сузить список до нужной тематики. В административном списке ресурсов, представленном на рисунке 9, видно, что рядом с каждой карточкой располагаются кнопки редактирования и удаления, однако в интерфейсе студента эти элементы полностью скрыты. Для проверки серверного ролевого разграничения выполняется попытка напрямую обратиться к защищённому API-эндпоинту создания новости с токеном студента через инструмент тестирования API; результат данного запроса показан на рисунке 13.

Рисунок 13 — Ответ API со статусом 403 при попытке создания новости с токеном пользователя роли STUDENT

Сервер возвращает ответ со статусом 403 Forbidden и сообщением об отсутствии необходимых прав, что подтверждает независимость серверной защиты от клиентских ограничений интерфейса. Результаты выполнения контрольного примера в совокупности подтверждают работоспособность всех ключевых компонентов системы: механизма аутентификации на основе JWT с сохранением токенов в SecureStore, ролевого разграничения доступа на уровне серверного API, публикации контента через формы мобильного приложения, отображения ленты новостей и каталога ресурсов с фильтрацией по категориям, а также детального просмотра публикаций. Время отклика API не превысило двухсот миллисекунд для стандартных операций чтения; интерфейс корректно адаптируется к роли текущего пользователя, скрывая или отображая элементы управления в зависимости от набора прав.
