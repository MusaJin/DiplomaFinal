## 1.6 Обоснование проектных решений по видам обеспечения

Завершающим разделом аналитической части является обоснование конкретных проектных решений по четырём видам обеспечения: техническому, информационному, программному и технологическому. Каждый из этих аспектов рассматривается в отдельности, однако в рамках общей концепции целостной системы, в которой проектные решения одного вида обеспечения неразрывно связаны с решениями по остальным видам. Цель данного раздела — не описание уже принятых решений, а их обоснование: демонстрация того, что выбранные инструменты, форматы, алгоритмы и архитектурные подходы являются оптимальными применительно к сформулированным ранее требованиям.

Техническое обеспечение системы охватывает совокупность аппаратных средств, необходимых для её функционирования, и включает клиентские устройства и серверную инфраструктуру. На стороне клиента минимальными требованиями являются смартфон под управлением Android версии 8.0 и выше или iOS версии 13 и выше с доступом к сети интернет. Выбор Android 8.0 в качестве нижней границы поддерживаемых версий обусловлен данными статистики мирового рынка: на момент разработки системы доля устройств с более ранними версиями Android составляла менее пяти процентов активной аудитории, и разработка под них потребовала бы значительных дополнительных усилий при ничтожном охвате. Аналогичная логика применяется к iOS 13: поддержка более ранних версий утратила практический смысл после прекращения обновлений безопасности для этих систем. Минимальные требования к объёму оперативной памяти устройства составляют два гигабайта, что соответствует характеристикам устройств среднего ценового сегмента, доступных большинству студентов.

Серверная инфраструктура для продуктивного развёртывания предполагает использование выделенного или виртуального сервера с операционной системой Linux (предпочтительно Ubuntu Server 20.04 LTS или более поздней версии) с объёмом оперативной памяти не менее двух гигабайт и частотой процессора не менее двух гигагерц. В условиях тестовой разработки и контрольного примера система функционирует на локальной машине разработчика, взаимодействие с которой со стороны Android-эмулятора осуществляется через специальный адрес 10.0.2.2, соответствующий loopback-интерфейсу хост-машины. Для организации сетевого взаимодействия в среде разработки применяется инструмент adb reverse, перенаправляющий запросы с порта эмулятора на соответствующий порт хост-машины. В продуктивном окружении сервер должен быть доступен по внешнему IP-адресу или доменному имени с действующим SSL-сертификатом для защиты транспортного уровня.

Информационное обеспечение системы определяется структурой хранимых данных и принципами их организации. Выбор реляционной СУБД PostgreSQL в качестве основного хранилища данных обоснован рядом её характеристик, непосредственно значимых для поставленной задачи. PostgreSQL обеспечивает строгое соблюдение свойств ACID (атомарность, согласованность, изолированность, долговечность) при выполнении транзакций, что критически важно для систем, в которых целостность данных является приоритетом. PostgreSQL поддерживает сложные типы данных, в том числе перечислимые типы (enum), которые используются в схеме для определения ролей пользователей и типов ресурсов. СУБД активно разрабатывается и поддерживается сообществом с 1996 года, располагает обширной документацией и широким набором инструментов администрирования.

Схема базы данных строится вокруг пяти основных сущностей. Таблица пользователей (users) хранит идентификатор, полное имя, адрес электронной почты, хеш пароля, роль, дату создания и обновления записи. Таблица категорий (categories) содержит идентификатор, наименование, тип категории (NEWS, RESOURCE, COMMON) и временны́е метки. Таблица новостей (news) включает идентификатор, заголовок, краткое описание, полный текст, URL изображения, признак опубликованности, ссылки на автора и категорию, а также дату публикации. Таблица образовательных ресурсов (resources) аналогична по структуре таблице новостей, но дополнена полем типа ресурса и полем содержимого (fileUrl или textContent в зависимости от типа). Структура данных спроектирована в соответствии с требованиями третьей нормальной формы, что исключает аномалии обновления и минимизирует дублирование данных.

Программное обеспечение системы делится на два независимых уровня — серверный и клиентский. Серверная часть построена на Node.js версии 18 LTS с использованием фреймворка Express 4 и языка TypeScript 5. Выбор Express обоснован его минималистичной философией: фреймворк предоставляет механизмы маршрутизации, обработки middleware и управления запросами/ответами, не навязывая конкретных архитектурных паттернов, что обеспечивает разработчику максимальную гибкость при проектировании структуры приложения. Альтернативой могло бы служить использование фреймворка NestJS, обеспечивающего более строгую структуру приложения и поддержку декораторов в стиле Angular. Однако для проекта данного масштаба дополнительный слой абстракций NestJS представлял собой неоправданное усложнение при минимальных преимуществах.

ORM Prisma выбрана в качестве инструмента взаимодействия с базой данных по нескольким причинам. Во-первых, Prisma обеспечивает автоматическую генерацию TypeScript-типов для всех сущностей схемы базы данных, что создаёт сквозную типизацию от уровня базы данных до уровня HTTP-ответа и позволяет обнаруживать несоответствия типов данных на этапе компиляции. Во-вторых, Prisma Migrate предоставляет декларативный механизм управления схемой базы данных: разработчик описывает целевое состояние схемы в Prisma Schema Language, а инструмент автоматически генерирует SQL-скрипты для перехода между состояниями. Это существенно упрощает процесс эволюции схемы данных в процессе разработки и сопровождения системы. В-третьих, Prisma Client генерирует идиоматичный TypeScript API для всех операций с данными, что делает код взаимодействия с базой данных читаемым и самодокументируемым.

Клиентская часть системы построена на React Native с использованием платформы Expo SDK 50. Управление состоянием приложения реализовано с помощью библиотеки Zustand — минималистичного решения для глобального состояния, лишённого бойлерплейта Redux при сохранении возможности подписки на отдельные фрагменты состояния. Выбор Zustand перед Redux обоснован значительно меньшим количеством шаблонного кода при аналогичных возможностях для задач данного масштаба: store аутентификации в Zustand занимает около тридцати строк кода против ста пятидесяти строк в типичной Redux-реализации. Навигация между экранами реализована с использованием React Navigation — де-факто стандарта навигации в экосистеме React Native, поддерживающего Stack-навигатор для вложенных переходов и BottomTabs-навигатор для основной структуры приложения. Сетевое взаимодействие с API реализовано через библиотеку Axios, обеспечивающую перехватчики (interceptors) для автоматической подстановки JWT-токена в заголовки запросов и обработки истекших токенов.

Технологическое обеспечение системы определяет порядок сбора, передачи, обработки и представления информации. Клиент-серверная архитектура с REST API выбрана как наиболее зрелый и широко поддерживаемый подход к организации взаимодействия между мобильным клиентом и серверной частью. REST предполагает использование стандартных HTTP-методов (GET для чтения, POST для создания, PUT/PATCH для обновления, DELETE для удаления) и унифицированных URL-адресов ресурсов, что делает API интуитивно понятным для разработчиков и упрощает его документирование и тестирование. Альтернативный подход на основе GraphQL был рассмотрен, однако отклонён: для системы с относительно простыми и предсказуемыми шаблонами запросов GraphQL не даёт ощутимых преимуществ, но существенно усложняет инфраструктуру и кривую обучения.

Аутентификация пользователей реализована на основе JSON Web Tokens — стандарта RFC 7519, определяющего компактный и самодостаточный формат передачи заявлений (claims) между сторонами. JWT-подход предпочтителен для мобильных приложений по сравнению с сессионной аутентификацией: токен содержит всю необходимую информацию о пользователе (идентификатор, роль) непосредственно в своей полезной нагрузке, что исключает необходимость обращения к базе данных для проверки сессии при каждом запросе. Это существенно снижает нагрузку на сервер при высокой частоте запросов и упрощает горизонтальное масштабирование. Access token имеет ограниченный срок действия (двадцать четыре часа) для снижения рисков, связанных с его перехватом, тогда как refresh token с более длительным временем жизни (семь суток) обеспечивает обновление пары токенов без требования повторного ввода пароля.
