## 1.3 Постановка задачи

Постановка задачи является центральным методологическим этапом проектирования информационной системы, на котором абстрактная потребность в автоматизации трансформируется в конкретную и верифицируемую систему требований. Качество этого перехода во многом определяет как техническую корректность реализации, так и практическую применимость готового продукта: система, разработанная без чёткой постановки задачи, рискует оказаться либо избыточной по функциональности, либо неспособной закрыть ключевые потребности целевой аудитории. В рамках настоящей работы постановка задачи строится на результатах аналитического этапа и формализует их в виде функциональных и нефункциональных требований к разрабатываемой системе.

Целью разрабатываемой системы является обеспечение централизованного, управляемого и надёжного информационного взаимодействия между участниками образовательного процесса посредством мобильного приложения. Система предназначена для автоматизации создания и публикации институциональных новостей с обеспечением их оперативной доступности для всех зарегистрированных пользователей, структурированного хранения и предоставления доступа к образовательным ресурсам различных типов, управления учётными записями с разграничением прав по ролевому принципу, а также классификации контента через систему настраиваемых категорий. Область применения системы ограничена рамками одного высшего учебного заведения и не предполагает интеграции с внешними образовательными платформами в рамках текущей версии продукта.

Функциональные требования определяют, что именно система обязана уметь делать, и задают критерии приёмки реализованного решения. Она должна обеспечивать регистрацию новых учётных записей с указанием полного имени, адреса электронной почты и пароля, аутентификацию пользователей с выдачей и хранением JWT-токенов, назначение роли каждой учётной записи — безопасность этого механизма обеспечивается хешированием паролей алгоритмом bcrypt и ограниченным сроком действия токенов доступа. Пользователи с правами администратора обязаны располагать полным инструментарием управления новостями: созданием публикации с заголовком, кратким описанием, полным текстом, опциональными изображением и категорией, редактированием ранее опубликованного материала и его удалением. Все зарегистрированные пользователи независимо от роли должны иметь доступ к списку опубликованных новостей с фильтрацией по категориям и к детальному представлению каждой публикации; хронологически список выстраивается в порядке убывания даты публикации. Преподаватели и администраторы наделяются правами на создание записей об образовательных ресурсах — с указанием наименования, описания, типа ресурса (LINK, FILE или TEXT) и соответствующих атрибутов, — а также на их редактирование и удаление; студентам предоставляется доступ к каталогу ресурсов с фильтрацией по категориям и типам и к детальному представлению с доступом к содержимому. Категориальный справочник, обеспечивающий единую таксономическую основу как для новостей, так и для ресурсов, управляется исключительно администратором — в его компетенцию входят создание категорий с указанием наименования и области применения (NEWS, RESOURCE или COMMON), их редактирование и удаление; набор категорий не является фиксированным и пополняется по мере развития информационной среды учреждения.

Нефункциональные требования задают качественные характеристики системы, не связанные непосредственно с её функциональным содержанием, но в равной мере определяющие практическую ценность разработки. В части производительности система должна обеспечивать время отклика на запросы к API не более пятисот миллисекунд при нормальной нагрузке, а список новостей должен загружаться и отображаться в течение двух секунд от момента открытия соответствующего экрана — показатели, соответствующие ожиданиям пользователей мобильных приложений, для которых задержка более двух-трёх секунд является значимым негативным фактором восприятия. Требования безопасности предусматривают хранение паролей исключительно в виде криптографических хешей (bcrypt, стоимостный фактор не менее десяти), передачу всех данных по защищённому каналу HTTPS в продуктивном окружении, использование JWT-токенов с ограниченным сроком действия — двадцать четыре часа для токена доступа и семь суток для токена обновления, — а также проверку роли пользователя на серверной стороне при каждом привилегированном запросе. Хранение токенов на клиентской стороне осуществляется в защищённом хранилище Expo SecureStore, что исключает их доступность через стандартные механизмы межприложенческого обмена данными.

Требование кроссплатформенности обязывает мобильный клиент корректно функционировать как на устройствах Android (версии 8.0 и выше), так и на устройствах iOS (версии 13 и выше). Масштабируемость системы определяется архитектурными решениями серверной части, которые должны допускать горизонтальное масштабирование без изменений в логике приложения: отсутствие состояния на уровне REST API и хранение сессионных данных в JWT-токенах, а не в серверной памяти, являются ключевыми решениями, обеспечивающими это свойство, — при возрастании числа пользователей сверх расчётных показателей достаточно запустить дополнительные экземпляры серверного приложения без вмешательства в его код.

Организация решения задачи строится на взаимодействии двух основных компонентов. Серверная часть на Node.js с Express обрабатывает HTTP-запросы от мобильного клиента, выполняет бизнес-логику и взаимодействует с базой данных через Prisma ORM; мобильный клиент на React Native с Expo взаимодействует с сервером через REST API, сохраняет аутентификационные токены в SecureStore и отображает полученный контент в соответствии с ролью пользователя. Связь между компонентами осуществляется через HTTP/HTTPS с использованием JSON как формата обмена данными — клиент-серверная архитектура обеспечивает чёткое разделение ответственности и создаёт основу для последующего проектирования программных модулей системы.
