## 2.2 Программное обеспечение задачи

Программное обеспечение разрабатываемой системы организовано по принципу двухуровневой клиент-серверной архитектуры, при которой серверная часть берёт на себя бизнес-логику, управление данными и взаимодействие с внешними сервисами, тогда как мобильный клиент отвечает за отображение информации, обработку пользовательского ввода и управление навигационным состоянием приложения. Разделение ответственности между уровнями реализовано через контракт REST API: клиент и сервер взаимодействуют исключительно посредством HTTP-запросов, что обеспечивает независимость их развёртывания и возможность независимого развития каждого уровня при сохранении неизменности API-контракта.

Дерево функций системы отражает иерархическую декомпозицию функциональности по ролям пользователей. На первом уровне декомпозиции выделяются три ветви: функции, доступные всем аутентифицированным пользователям независимо от роли; функции, специфичные для роли TEACHER; функции, специфичные для роли ADMIN. К числу общих функций относятся: аутентификация, просмотр ленты новостей с фильтрацией по категориям, просмотр каталога образовательных ресурсов. Ветвь TEACHER дополнительно включает создание, редактирование и удаление собственных образовательных ресурсов. Ветвь ADMIN включает полное управление новостями, ресурсами и категориями.

Сценарий диалога пользователя с системой начинается с экрана входа, который является единственным экраном, доступным неаутентифицированному пользователю. Интерфейс экрана входа содержит два поля ввода — адрес электронной почты и пароль — и кнопку подтверждения, что наглядно представлено на рисунке 1.

Рисунок 2 — Главный экран приложения

Из любого списка пользователь может перейти на экран детального просмотра, который является частью Stack-навигатора, вложенного в BottomTabs-навигатор. Набор вкладок нижней навигационной панели варьируется в зависимости от роли: пользователям с ролью ADMIN отображается дополнительная вкладка «Управление», которая скрыта для ролей TEACHER и STUDENT.

Архитектура серверной части организована по модульному принципу с разделением на слои маршрутизации, контроллеров, сервисов и доступа к данным через Prisma Client. Корневой файл приложения инициализирует Express-приложение, подключает middleware для обработки JSON, настраивает CORS, регистрирует маршрутизаторы модулей и запускает HTTP-сервер. Каждый функциональный модуль — auth, news, resources, categories — организован как отдельный набор файлов с собственным Express Router. Маршруты REST API организованы в соответствии с REST-соглашениями: группа /api/auth включает POST /register, POST /login и POST /refresh; группа /api/news — GET /, GET /:id, POST /, PUT /:id и DELETE /:id с разграничением прав по роли ADMIN; группа /api/resources — аналогичную структуру с расширением прав создания для TEACHER и ADMIN; группа /api/categories — CRUD-операции, ограниченные ролью ADMIN.

Middleware авторизации реализован в виде двух функций Express-middleware. Первая, authenticate, проверяет наличие JWT в заголовке Authorization (схема Bearer), верифицирует подпись токена с использованием секретного ключа из переменных окружения и помещает декодированную полезную нагрузку (userId, role) в объект запроса; при отсутствии или недействительности токена middleware немедленно возвращает ответ со статусом 401. Вторая функция, authorize, принимает массив допустимых ролей и проверяет, входит ли роль текущего пользователя в этот массив; при несоответствии возвращается ответ со статусом 403.

Навигационная структура приложения строится на двух уровнях. Корневой уровень представлен Stack-навигатором, который управляет переходом между экраном аутентификации и основным контентом. Основная навигация реализована через BottomTabNavigator с тремя вкладками, каждая из которых является дочерним Stack-навигатором, что позволяет поддерживать историю навигации в каждой вкладке независимо. Вкладка «Новости» открывает список публикаций с горизонтальными фильтрами по категориям, представленный на рисунке 3.

Рисунок 6 — Экран детального просмотра образовательного ресурса

Интерфейс детального просмотра адаптируется к типу ресурса: для LINK отображается кнопка «Открыть ссылку», открывающая URL во встроенном браузере; для FILE — кнопка загрузки; для TEXT — текстовое содержимое, хранящееся непосредственно в базе данных.

Zustand store аутентификации хранит объект текущего пользователя (id, fullName, email, role), флаг загрузки и ссылки на функции логина, выхода и инициализации. При инициализации приложения store считывает сохранённые токены из Expo SecureStore и, если токен действителен, восстанавливает сессию пользователя без требования повторного ввода учётных данных. Функция выхода очищает SecureStore и обнуляет состояние хранилища, после чего навигатор автоматически перенаправляет пользователя на экран входа.

Модуль сервиса API реализован как набор функций, использующих экземпляр Axios с настроенными interceptors. Interceptor запроса автоматически добавляет заголовок Authorization: Bearer <token> ко всем исходящим запросам, извлекая актуальный access token из SecureStore. Interceptor ответа перехватывает ответы со статусом 401 и автоматически выполняет попытку обновления пары токенов через endpoint /api/auth/refresh; при успешном обновлении повторяет исходный запрос с новым токеном; при неудаче — инициирует выход пользователя из системы.

Описание ключевых программных модулей начинается с модуля NewsListScreen, отвечающего за отображение ленты новостей. При монтировании компонента выполняется параллельная загрузка категорий и новостей через соответствующие API-методы; результат загрузки сохраняется в локальном состоянии компонента; при выборе категории фильтрации выполняется повторный запрос к API с параметром categoryId; список новостей отображается через FlatList с поддержкой обновления через pull-to-refresh. Модуль AdminNewsListScreen, доступный пользователям с ролью ADMIN, отображает полный список новостей включая неопубликованные черновики, что отражено на рисунке 7.
