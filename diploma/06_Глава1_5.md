## 1.5 Анализ существующих разработок, выбор и обоснование стратегии автоматизации

Перед принятием решения о самостоятельной разработке информационной системы необходимо провести объективный анализ существующих программных продуктов, потенциально решающих аналогичные задачи. Изучение рынка готовых решений выполняет двойную функцию: с одной стороны, оно позволяет установить, не существует ли продукта, который в полной мере удовлетворяет сформулированным требованиям и может быть внедрён без значительной доработки; с другой стороны, анализ чужих реализаций позволяет выявить отраслевые стандарты, удачные архитектурные и интерфейсные решения, а также типичные ошибки и ограничения, которых следует избегать при собственной разработке. Рассмотрению подлежат как универсальные образовательные платформы, так и решения, ориентированные непосредственно на коммуникацию внутри учебного заведения.

Microsoft Teams для образования представляет собой часть экосистемы Microsoft 365 Education и реализует широкий спектр функций корпоративного общения, адаптированных для академической среды: организацию виртуальных классов, совместное редактирование документов, проведение видеоконференций, управление заданиями и их оценивание. Продукт обладает зрелой мобильной клиентской частью для Android и iOS и теоретически может использоваться для распространения институциональных новостей через механизм каналов и объявлений. Вместе с тем применение Teams в качестве инструмента рассылки новостей для всего университета сопряжено с рядом серьёзных ограничений. Прежде всего, система требует лицензионной подписки на Microsoft 365 Education, что создаёт постоянные финансовые обязательства. Второй проблемой является избыточная функциональность: для пользователя, которому нужно лишь просмотреть последние новости вуза или скачать методичку, необходимость ориентироваться в полнофункциональном корпоративном инструменте является существенным барьером. Наконец, кастомизация Teams под конкретную организационную структуру российского вуза требует значительных административных ресурсов и компетенций, которыми располагают далеко не все учреждения.

Moodle Mobile — мобильное приложение для системы управления обучением Moodle с открытым исходным кодом — представляет собой один из наиболее распространённых инструментов в сфере дистанционного образования. Moodle обеспечивает публикацию учебных материалов, управление курсами, тестирование, форумы и систему уведомлений. Мобильный клиент Moodle Mobile обеспечивает доступ ко всем этим возможностям с мобильного устройства. Тем не менее применительно к задаче рассылки институциональных новостей Moodle обнаруживает принципиальное несоответствие: платформа спроектирована как инструмент управления учебным процессом, а не как информационный портал учреждения, и задача оперативного оповещения всех пользователей решается в ней опосредованно — через форумы, ресурсы курсов или плагины, требующие отдельной настройки. Пользовательский интерфейс Moodle Mobile нередко критикуется за перегруженность и низкую интуитивность, что снижает вероятность регулярного использования приложения студентами вне контекста активных учебных задач.

ВКонтакте для образования и приложение «VK Мессенджер» широко используются российскими университетами для неформального информирования студентов через группы и публичные страницы. Охват аудитории платформы ВКонтакте среди российской молодёжи чрезвычайно высок, что делает её привлекательным каналом коммуникации. Однако использование социальной сети в качестве официального институционального канала влечёт ряд принципиальных проблем. Администрация вуза не контролирует правила модерации и алгоритмы ранжирования контента платформы, что означает, что важное официальное объявление может быть скрыто алгоритмом в ленте за несколькими развлекательными публикациями. Публикации в социальной сети лишены юридического статуса официального документа. Кроме того, хранение персональных данных студентов на серверах частной коммерческой организации порождает вопросы соответствия требованиям законодательства о персональных данных. Все перечисленные факторы исключают ВКонтакте из числа приемлемых решений для официального информационного обеспечения университета.

Яндекс.Учебник представляет собой образовательную платформу, ориентированную преимущественно на общеобразовательные школы и не предназначенную для использования в высшей школе. Функциональность платформы сосредоточена на автоматизированной проверке домашних заданий и формировании учебных траекторий для школьников, что делает её неприменимой в контексте информационного обеспечения высшего учебного заведения. Её включение в обзор обусловлено широкой известностью продукта в российском образовательном пространстве и необходимостью явного разграничения между инструментами для высшей и общеобразовательной школы.

Сравнительный анализ рассмотренных решений по ключевым критериям позволяет сформировать системное представление об их пригодности для решения поставленной задачи. По критерию соответствия функциональных возможностей сформулированным требованиям ни одно из рассмотренных решений не обеспечивает оптимального сочетания функций публикации новостей, управления образовательными ресурсами, ролевого разграничения и push-уведомлений в едином лаконичном интерфейсе. По критерию стоимости владения Microsoft Teams и ряд других корпоративных платформ требуют постоянных лицензионных расходов, тогда как Moodle с открытым исходным кодом предполагает значительные затраты на развёртывание, настройку и сопровождение инфраструктуры. По критерию пользовательского опыта все рассмотренные решения уступают специализированному мобильному приложению в части интуитивности интерфейса и скорости доступа к нужной функции, поскольку разработаны для значительно более широкого спектра задач. По критерию соответствия организационной специфике российских вузов лишь отечественные платформы не создают проблем с хранением данных и соответствием законодательству, однако они не обеспечивают необходимой функциональности.

На основании проведённого анализа был сделан вывод о целесообразности самостоятельной разработки специализированного мобильного приложения. Стратегия самостоятельной разработки предпочтительна по следующим основаниям. Во-первых, специализированное решение может быть спроектировано в точном соответствии с функциональными требованиями без избыточной функциональности, ориентированной на иные сценарии использования. Во-вторых, полный контроль над исходным кодом обеспечивает возможность развития и модификации системы силами собственных разработчиков или сторонних исполнителей без зависимости от решений вендора. В-третьих, отсутствие лицензионных платежей и возможность развёртывания на собственной инфраструктуре минимизируют долгосрочные издержки владения. В-четвёртых, разработка нового продукта на современном технологическом стеке обеспечивает его соответствие актуальным стандартам в области мобильного UX и позволяет с самого начала учесть специфику целевой аудитории.

Выбор React Native в качестве технологии разработки мобильного клиента обоснован несколькими взаимодополняющими аргументами. React Native является кроссплатформенным фреймворком, обеспечивающим создание приложений для Android и iOS из единой кодовой базы на языке JavaScript/TypeScript. Это принципиально отличает его от нативной разработки, при которой поддержка двух платформ требует ведения двух независимых проектов на разных языках программирования (Swift/Objective-C для iOS, Kotlin/Java для Android). В условиях ограниченного бюджета разработки и команды из одного-двух разработчиков кроссплатформенный подход является единственным практически реализуемым вариантом создания полноценного мобильного продукта для обеих доминирующих платформ. Альтернативный кроссплатформенный фреймворк Flutter от Google также рассматривался в качестве кандидата, однако React Native обладает рядом конкурентных преимуществ: более зрелая экосистема JavaScript-библиотек, значительно большее сообщество разработчиков, более тесная интеграция с веб-технологиями (что упрощает привлечение веб-разработчиков к проекту) и прямое использование нативных компонентов платформы без необходимости самостоятельной отрисовки UI.

Платформа Expo SDK 50, используемая поверх React Native, обеспечивает ряд критически важных возможностей, недоступных в «голом» React Native без значительного объёма нативной настройки. Expo предоставляет готовую интеграцию с системой push-уведомлений через сервис Expo Notifications, защищённое хранилище учётных данных (SecureStore), работу с изображениями (ImagePicker), файловой системой и рядом других нативных возможностей устройства — всё это в виде JavaScript API без необходимости написания нативного кода на Swift или Kotlin. Expo также обеспечивает инструментарий для разработки и тестирования — Expo Go — позволяющий запускать приложение на физическом устройстве без сборки APK, что существенно ускоряет цикл разработки. Совокупность этих возможностей делает Expo оптимальным выбором для проекта данного масштаба.

Для серверной части выбран стек Node.js с фреймворком Express и языком TypeScript. Node.js обеспечивает асинхронную, событийно-ориентированную обработку запросов, что делает его эффективным для REST API с высокой частотой параллельных запросов и при этом относительно небольшой вычислительной нагрузкой. Использование JavaScript/TypeScript как единого языка для клиентской и серверной части позволяет переиспользовать типы данных и утилиты между слоями приложения, снижая риск несоответствия между форматами данных, которыми обмениваются клиент и сервер. TypeScript добавляет статическую типизацию, которая выявляет потенциальные ошибки на этапе компиляции, а не в процессе выполнения, что особенно ценно при разработке API с формальным контрактом. ORM Prisma предоставляет типобезопасный интерфейс для работы с базой данных PostgreSQL, генерируя TypeScript-типы для всех сущностей схемы и обеспечивая автоматизированное применение миграций при изменении структуры данных.
