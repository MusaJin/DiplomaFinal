## 2.3 Технологическое обеспечение задачи

Технологическое обеспечение системы описывает совокупность технологических процессов, через которые проходит информация от момента её создания до представления конечному пользователю. В отличие от программного обеспечения, фиксирующего структуру кода и архитектуру компонентов, технологическое обеспечение рассматривает систему с динамической точки зрения — как последовательность взаимосвязанных операций над данными, инициируемых событиями в жизненном цикле приложения. К числу основных технологических процессов системы относятся: регистрация и аутентификация пользователей, технологический цикл публикации новости, цикл создания образовательного ресурса, а также технологический процесс получения и отображения контента мобильным клиентом.

Технологический процесс регистрации пользователя начинается с заполнения формы регистрации в мобильном клиенте, содержащей поля полного имени, адреса электронной почты и пароля. После подтверждения ввода мобильный клиент формирует HTTP POST-запрос к эндпоинту /api/auth/register с телом в формате JSON, содержащим введённые данные. Сервер выполняет последовательность операций: валидация структуры тела запроса, проверка уникальности адреса электронной почты в базе данных, хеширование пароля с использованием алгоритма bcrypt с коэффициентом стойкости 10, создание записи пользователя в базе данных. При успешном завершении сервер возвращает ответ со статусом 201 и объектом созданного пользователя (без поля пароля). Клиент автоматически выполняет вход пользователя, вызывая эндпоинт /api/auth/login с введёнными учётными данными. Технологический процесс аутентификации предполагает отправку учётных данных (email, password) на эндпоинт /api/auth/login, интерфейс которого показан на рисунке 1. Сервер извлекает запись пользователя из базы данных по email, проверяет соответствие введённого пароля хешу с использованием bcrypt.compare, генерирует пару токенов (access token с временем жизни 24 часа, refresh token с временем жизни 7 суток) посредством библиотеки jsonwebtoken и возвращает их клиенту вместе с объектом пользователя. Мобильный клиент сохраняет полученные токены в Expo SecureStore — защищённом хранилище, использующем Keychain (iOS) или Keystore (Android) для шифрования данных на уровне операционной системы. После завершения инициализации пользователь оказывается на главном экране приложения, отражённом на рисунке 2, набор вкладок которого варьируется в зависимости от присвоенной роли. Пользователям с ролью ADMIN на главном экране отображается дополнительная вкладка «Управление», недоступная ролям TEACHER и STUDENT, что реализует разграничение пользовательского интерфейса в соответствии с уровнем прав. Этот механизм опирается на значение роли, хранящееся в Zustand store, и применяется на стороне клиента исключительно для улучшения пользовательского опыта: основная защита обеспечивается на уровне серверного API через middleware авторизации.

Технологический процесс публикации новости инициируется администратором на экране создания или редактирования новости, форма которого представлена на рисунке 8. В форме заполняются обязательные поля: заголовок, краткое описание и полный текст публикации; опционально выбирается категория из выпадающего списка и указывается URL изображения. После нажатия кнопки сохранения мобильный клиент выполняет POST-запрос к /api/news (при создании) или PUT-запрос к /api/news/:id (при редактировании) с данными новости в теле запроса. Middleware authenticate проверяет JWT-токен из заголовка запроса; middleware authorize проверяет, что роль пользователя равна ADMIN. Контроллер создаёт или обновляет запись новости в базе данных через Prisma Client с установкой признака published=true и фиксацией текущей даты в поле publishedAt. Сервер возвращает клиенту объект созданной или обновлённой новости со статусом 201, после чего клиент выполняет переход на экран административного списка новостей, показанного на рисунке 7, где новая запись отображается первой с меткой «Опубликовано».

Технологический процесс получения и отображения ленты новостей мобильным клиентом начинается с монтирования компонента NewsListScreen. В методе useEffect компонент параллельно запрашивает два эндпоинта: GET /api/categories?type=NEWS для загрузки доступных фильтров и GET /api/news для загрузки первой страницы новостей. Ответ сервера содержит массив объектов, включающих вложенные объекты автора и категории, что позволяет отобразить все необходимые данные без дополнительных запросов. При нажатии на карточку выполняется навигационный переход на экран NewsDetailScreen с передачей идентификатора новости в параметрах маршрута; этот экран запрашивает полный объект через GET /api/news/:id и отображает изображение, метаданные автора, дату, краткое описание в выделенном блоке и полный текст, что показано на рисунке 4. Отображение полного текста публикации на отдельном экране позволяет избежать перегрузки списковых представлений и обеспечивает удобное чтение материалов любого объёма. Компонент FlatList, применяемый для рендеринга ленты, поддерживает обновление через pull-to-refresh, при котором инициируется повторный запрос к API с теми же активными параметрами фильтрации.

Технологический процесс управления образовательными ресурсами для преподавателя включает переход на экран создания ресурса через кнопку в шапке каталога, заполнение формы с указанием названия, описания, типа ресурса и соответствующего содержимого (URL или текст), а также выбор категории. Каталог ресурсов с применёнными фильтрами и цветовой кодировкой типов представлен на рисунке 5. После сохранения сервер создаёт запись в таблице resources с привязкой к идентификатору текущего пользователя в качестве автора. Редактирование доступно автору ресурса и администратору: при переходе к редактированию форма предзаполняется текущими значениями записи, загружаемыми по идентификатору через GET /api/resources/:id. При удалении записи выполняется DELETE-запрос к соответствующему эндпоинту; после успешного ответа от сервера список ресурсов обновляется локально без полной перезагрузки данных. Организация технологии сбора, передачи и обработки информации в целом подчиняется единой схеме: ввод через формы React Native, передача по HTTPS в формате JSON с Bearer-токеном в заголовке, серверная обработка через цепочку middleware и Prisma Client, возврат JSON-ответа клиенту с соответствующим HTTP-статусом.

Технологический процесс обновления токенов аутентификации реализован прозрачно для пользователя через interceptor Axios на стороне клиента. При получении ответа с кодом 401 interceptor проверяет, не является ли сам запрос запросом обновления токена во избежание бесконечного цикла; если нет — извлекает refresh token из SecureStore, выполняет POST-запрос к /api/auth/refresh, сохраняет новую пару токенов и повторяет исходный запрос с обновлённым access token. Если запрос обновления токена возвращает ошибку — пользователь перенаправляется на экран входа, представленный на рисунке 1. Этот механизм обеспечивает непрерывную работу приложения в течение времени жизни refresh token (7 суток) без видимых прерываний для пользователя.

Технологический процесс управления ролевым доступом реализован на двух уровнях. На уровне сервера каждый защищённый маршрут включает в цепочку обработчиков функцию authorize с указанием допустимых ролей; проверка роли производится по значению, извлечённому из JWT-токена middleware authenticate: поскольку токен подписан серверным секретом, его содержимое не может быть подделано клиентом. На уровне мобильного клиента роль текущего пользователя хранится в Zustand store и используется для условного отображения элементов интерфейса. Такой подход обеспечивает защиту на уровне API как основного защитного механизма при одновременном улучшении пользовательского опыта за счёт отсутствия недоступных элементов управления в интерфейсе.
