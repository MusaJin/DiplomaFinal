## [Слайд 2 — Актуальность]

Переходим к актуальности.

Современные студенты получают информацию преимущественно через смартфон — более 80% обращаются к учебному контенту именно с мобильных устройств. Между тем большинство вузов, включая СКГА, по-прежнему использует разрозненные каналы: официальный сайт, электронную почту, группы в мессенджерах. Это приводит к дублированию, задержкам и невозможности гарантировать, что студент получит важную информацию вовремя.

Таким образом, разработка единого мобильного канала является не факультативным улучшением, а необходимым условием нормального функционирования информационного пространства современного университета.

## [Слайд 3 — Объект, предмет и цель]

Объектом исследования являются **процессы информационного взаимодействия** участников образовательного процесса.

Предмет — **технологии разработки кросс-платформенных мобильных приложений** на базе React Native.

Цель — **разработать и реализовать** мобильное приложение для централизованной публикации и рассылки новостей и образовательных ресурсов СКГА с разграничением прав доступа по ролям.

## [Слайд 4 — Задачи]

Для достижения цели были сформулированы шесть задач:
провести анализ предметной области,
спроектировать клиент-серверную архитектуру и схему базы данных,
реализовать серверную часть,
разработать мобильный клиент,
внедрить JWT-аутентификацию с ролевым разграничением доступа,
и провести тестирование системы с оформлением документации.

## [Слайд 5 — Стек технологий]

Система реализована на современном стеке технологий.

**Мобильный клиент** — React Native с Expo SDK 50, TypeScript, Zustand для управления состоянием и Expo Router для навигации.

**Серверная часть** — Node.js, Express, TypeScript и Prisma ORM.

**Хранилище данных** — PostgreSQL 16. Авторизация построена на основе JWT-токенов, учётные данные хранятся в Expo SecureStore.

---

## [Слайд 6 — Бизнес-процессы]

Система поддерживает три роли пользователей.

**Студент** — просматривает новости и учебные материалы, получает уведомления.

**Преподаватель** — публикует и редактирует образовательные ресурсы.

**Администратор** — управляет всем контентом, категориями и пользователями.

Взаимодействие обеспечивается через единый REST API на Node.js и PostgreSQL.

---

## [Слайд 7 — Схема данных]

База данных включает пять таблиц: **User, News, Resource, Notification и Category**.

Схема спроектирована в третьей нормальной форме. Ролевая модель реализована через PostgreSQL enum с тремя значениями: STUDENT, TEACHER, ADMIN. Ссылочная целостность обеспечивается внешними ключами с политиками ON DELETE RESTRICT и SET NULL.

---

## [Слайды 8–16 — Экранные формы]

Разработаны девять экранных форм.

**Экран авторизации** обеспечивает вход по email и паролю, регистрацию и сохранение токенов в SecureStore.

**Лента новостей** отображает карточки публикаций с фильтрацией по категориям и поддержкой pull-to-refresh.

**Просмотр статьи** — полный текст, метаданные автора, дата и возможность поделиться.

**Образовательные ресурсы** — каталог с цветовой кодировкой типов: ссылка, файл или текст.

**Просмотр документа** — встроенный просмотр или загрузка файла.

**Профиль** — личные данные, смена пароля, выход из аккаунта.

**Уведомления** — список системных сообщений с отметкой о прочтении.

**Настройки** — выбор темы, управление уведомлениями, очистка кэша.

**Административная панель** — полное управление контентом: публикация новостей, загрузка ресурсов, управление пользователями и рассылка уведомлений.

## [Слайд 17 — Выводы]

В ходе работы достигнуты следующие результаты.

Проведён анализ предметной области и изучены программные аналоги — Microsoft Teams, Moodle и ВКонтакте — которые не обеспечивают оптимального сочетания функций при приемлемых затратах.

Спроектирована клиент-серверная архитектура, реализован REST API и мобильный клиент с девятью экранными формами.

Внедрена JWT-аутентификация, ролевое разграничение и защищённое хранение данных.

Работоспособность системы подтверждена контрольным примером: время отклика API не превысило **200 миллисекунд**, попытка доступа к привилегированному эндпоинту с токеном студента корректно возвращает ответ **403 Forbidden**.

Система готова к развёртыванию в реальной среде вуза и может быть расширена модулями комментирования, расписания и офлайн-режима.
