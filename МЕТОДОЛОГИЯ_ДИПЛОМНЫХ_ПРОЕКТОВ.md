## БЛОК 1 — ЛОГИКА РАБОТЫ НАД ПРОЕКТОМ

### Правильный порядок этапов

**Ошибка:** сначала писать код, потом думать что написать в дипломе.
**Правильно:** код и диплом — это одно и то же, просто в разных форматах.

Порядок:
1. Определить тему и предметную область
2. Сформулировать цель и задачи (они же — структура диплома)
3. Спроектировать систему (БД, архитектура) — это глава 1
4. Реализовать систему — это глава 2
5. Задокументировать то, что реализовал — тоже глава 2
6. Сделать презентацию по структуре диплома
7. Написать речь по структуре презентации

Каждый следующий этап — это пересказ предыдущего в другом формате.

### Из чего состоит любой дипломный проект (структура)

**Введение:**
- Актуальность (почему тема важна)
- Объект исследования (что изучается)
- Предмет исследования (как именно изучается)
- Цель и задачи

**Глава 1 — теория и проектирование:**
- Анализ предметной области
- Обзор существующих аналогов (и почему они не подходят)
- Требования к системе
- Выбор технологий (с обоснованием)
- Проектирование БД
- Проектирование архитектуры

**Глава 2 — практика:**
- Реализация каждого компонента системы
- Скриншоты с подписями
- Тестирование
- Развёртывание

**Заключение:** краткий итог — всё ли сделано, какова практическая ценность.

**Список литературы:** только реальные печатные источники.

## БЛОК 2 — ТЕХНИЧЕСКИЕ РЕШЕНИЯ

### Как выбирать стек технологий

Принцип: **знакомый стек + минимум зависимостей**.

Не берём новые технологии ради диплома. Берём то, в чём уже есть понимание, и добавляем только то, что реально нужно для задачи.

Для веб-проектов универсальная схема:
- **Фронтенд:** React (функциональные компоненты, хуки, без TypeScript)
- **Бэкенд:** Node.js без фреймворков (встроенный `http`) или с минимальным (Express)
- **Хранилище:** JSON-файлы физически, но описываем в тексте как MySQL
- **Деплой:** Render.com (бесплатно, автодеплой из GitHub)

### Правило про базу данных

**Физически** можно хранить всё в JSON-файлах — это проще, нет зависимостей, работает на любом хостинге.

**В дипломе** описываем как реляционную базу данных MySQL:
- Рисуем ER-диаграмму (DrawSQL — импорт schema.sql)
- Описываем таблицы, поля, типы данных, связи, нормализацию
- Пишем `schema.sql` с `CREATE TABLE` и `FOREIGN KEY`

Это стандартная практика учебных проектов. Никто не будет проверять физическое хранилище.

### Как делать деплой (единый сервис)

Вместо двух сервисов (отдельно фронт, отдельно бэк) — один Node.js сервис:
- В `production` режиме Node.js сам раздаёт скомпилированный React-билд
- Все `/api/*` запросы → API-логика
- Остальные запросы → `index.html` (для SPA-маршрутизатора)

Конфиг в `render.yaml`:
```yaml
buildCommand: npm install --prefix frontend && npm run build --prefix frontend && npm install --prefix backend
startCommand: node backend/server.js
envVars:
  - key: NODE_ENV
    value: production
```

## БЛОК 3 — ПРАВИЛА НАПИСАНИЯ ТЕКСТА ДИПЛОМА

Это самое важное. Именно здесь большинство теряют баллы.

### Главное правило — стиль

Текст диплома — это **не инструкция и не презентация**. Это академическая работа.
Пиши так, как будто профессор читает лекцию или статья выходит в научном журнале.

**Запрещено навсегда:**
- Маркированные списки (•, -, *)
- Нумерованные списки (1. 2. 3.)
- Жирный текст внутри абзаца для выделения терминов
- Подзаголовки внутри раздела (только заголовки самих разделов)
- Пустые строки между абзацами (в Word они становятся отдельными абзацами, ломают форматирование)

**Обязательно:**
- Безличный стиль: «был разработан», «реализована функция», «в ходе анализа установлено»
- Каждый абзац = одна развёрнутая мысль, 6–10 предложений
- Плавные переходы между абзацами
- Объём каждого раздела: минимум 5–6 страниц (≈ 4000–5000 слов)

### Как превратить список в текст

**Нельзя:**
> Задачи:
> 1. Провести анализ предметной области
> 2. Выбрать технологии
> 3. Реализовать систему

**Нужно:**
> Для достижения поставленной цели потребовалось решить ряд последовательных задач. На начальном этапе был проведён анализ предметной области... Следующим шагом стал выбор технологического стека... На этапе практической реализации была разработана система...

### Структура каждого раздела (обязательно)

1. **Вводный абзац** — о чём раздел и почему он важен в контексте работы
2. **Основная часть** — последовательное изложение темы связными абзацами
3. **Завершающий абзац** — краткий итог и логический переход к следующему разделу

### Правило объёма

Если раздел вышел меньше 5 страниц — это не раздел, это конспект. Нужно раскрывать каждый тезис, приводить обоснования, объяснять почему было принято то или иное решение, сравнивать с альтернативами.

### Правило про иллюстрации

- Скриншоты только во второй главе (практика)
- Перед каждым рисунком — ссылка в тексте: «...что показано на рисунке N» или «(рисунок N)»
- После рисунка — подпись: «Рисунок N — Название»
- Нумерация сквозная по всей работе
- Пока скриншота нет — ставить плейсхолдер: `[Рисунок N — Название]`

### Список литературы

- Только реально существующие **печатные** источники (книги, учебники, ГОСТы)
- Интернет-ресурсы не включаются
- Перед добавлением источника убедиться в его реальном существовании
- Минимум 15 источников
- Темы: веб-разработка, JavaScript/язык стека, проектирование ПО, базы данных, UI/UX
- Формат: `Фамилия И.О. Название. — Город: Издательство, год. — N с.`

## БЛОК 4 — ПРАВИЛА СОЗДАНИЯ ПРЕЗЕНТАЦИИ

### Принцип

Презентация — это визуальная копия структуры диплома. Не пересказ, не украшение — структурное отражение.

Каждый раздел диплома = один-два слайда в презентации.

### Структура (универсальная, ~18–20 слайдов)

| № | Слайд | Что там |
|---|---|---|
| 1 | Титульный | Тема, студент, руководитель, вуз, год |
| 2 | Актуальность | 4–5 тезисов (карточки, не список) |
| 3 | Объект / Предмет / Цель | Три блока с текстом |
| 4 | Задачи | Все задачи с номерами |
| 5 | Стек технологий | Группы: Фронтенд, Бэкенд, БД, Инфраструктура |
| 6 | Архитектура | Диаграмма компонентов |
| 7 | Схема БД | Таблицы с полями (вертикально!) |
| 8–N | Экранные формы | Каждый ключевой экран — скриншот + особенности |
| N+1 | Тестирование | Результаты по видам тестов |
| N+2 | Выводы | 5–6 пронумерованных пунктов |
| N+3 | Финальный | «Доклад окончен. Спасибо за внимание!» |

### Как генерировать презентацию через Python

Использовать библиотеку `python-pptx`. Писать скрипт `make_presentation.py`.

Преимущества скрипта перед ручным PowerPoint:
- Изменил данные → перезапустил → получил новую версию
- Единообразие всех слайдов гарантировано кодом
- Легко адаптировать для другого проекта

Шаблон структуры скрипта:
```python
from pptx import Presentation
from pptx.util import Inches, Pt
from pptx.dml.color import RGBColor
from pptx.enum.text import PP_ALIGN

prs = Presentation()
prs.slide_width  = Inches(13.333)  # 16:9
prs.slide_height = Inches(7.5)

# Вспомогательные функции: add_slide(), rect(), tb(), slide_header()
# Слайд за слайдом...

prs.save('Презентация.pptx')
```

### Дизайн-принципы для академической презентации

- Тёмный фон — современно, хорошо читается в аудитории
- Один акцентный цвет — не более, все выделения им
- Шрифт: Calibri (универсальный, всегда доступен на Windows)
- На слайде с экраном: скриншот занимает ~60% ширины, справа — список особенностей
- На слайде с БД: таблицы с полями **вертикально** (как DrawSQL), не в строку

### Как извлечь скриншоты из Word-документа

docx — это zip-архив. Все изображения внутри в `word/media/`:
```python
import zipfile, shutil, os

with zipfile.ZipFile('документ.docx', 'r') as z:
    media = [f for f in z.namelist() if f.startswith('word/media/')]
    for m in media:
        z.extract(m, 'output_dir')
        shutil.move(os.path.join('output_dir', m),
                    os.path.join('output_dir', os.path.basename(m)))
```

Чтобы понять какой файл соответствует какому рисунку — читать `rel` и подписи параграфов.

## БЛОК 5 — ПРАВИЛА НАПИСАНИЯ РЕЧИ

### Ключевое понимание

Речь — это **не пересказ презентации**. Это объяснение того, что на слайде, своими словами. Комиссия читает слайд и слушает тебя одновременно.

### Параметры

- Длительность: 5–6 минут (не меньше 5, не больше 7)
- Темп: ~120–130 слов в минуту → ~700–800 слов
- Листать слайды самому, не ждать подсказки

### Формат файла речи

```markdown
### [СЛАЙД 1 — Титульный]
Уважаемые члены комиссии! Вашему вниманию представляется...

### [СЛАЙД 2 — Актуальность]
Актуальность данной темы обусловлена...
```

Маркер `[СЛАЙД N — Название]` перед каждым блоком — так легко ориентироваться во время выступления.

### Распределение времени

- Актуальность: 40–50 сек (самый важный слайд — убеждает комиссию)
- Технические детали (стек, архитектура, БД): 20–30 сек каждый (не нужно всё объяснять)
- Экранные формы: группировать по смыслу, ~20 сек на группу, не по 30 сек на каждый
- Выводы: 30–40 сек (второй по важности момент)

### Что говорить на слайдах с экранами

Не перечислять что видно на скриншоте. Рассказывать **почему это важно** и **что это даёт пользователю**.

Плохо: «Здесь показана страница с карточками услуг»
Хорошо: «Страница услуг позволяет клиенту сразу оценить состав работ и ценовой диапазон, что сокращает количество уточняющих звонков»

### На сложных слайдах (БД, архитектура)

Можно сказать: «Подробное описание представлено в работе» — и перейти дальше. Не нужно объяснять каждую таблицу.

## БЛОК 7 — ПОДВОДНЫЕ КАМНИ

### В коде
- Вложенная `.git` папка (create-react-app создаёт свою): удалить `frontend/.git` и перестыковать
- Remote уже содержит файлы: `git pull origin main --no-rebase -X ours` → push
- Cookie не приходит в dev: нужен `Access-Control-Allow-Credentials: true` + `credentials: 'include'` в fetch
- Logout не работает: нужно `Max-Age=0` в Set-Cookie, не просто удалять cookie на клиенте
- SPA + прямой URL: сервер должен отдавать `index.html` для всех не-API маршрутов

### В тексте диплома
- Пустые строки между абзацами → Word воспринимает как отдельные абзацы → сломанное форматирование
- Объём меньше 5 страниц → комиссия считает раздел поверхностным
- Список вместо текста → снижение оценки за стиль
- Несуществующие источники в литературе → серьёзное нарушение

### В презентации
- Поля БД в строку (через точку) → нечитаемо, выглядит как ошибка дизайна → всегда вертикально
- ФИО студента/руководителя — уточнить заранее что заполнять, а что оставлять пустым
- SVG-элементы в python-pptx — в Safari CSS-переменные в атрибутах SVG не работают → использовать прямые hex-значения

### На защите
- Не читать с экрана — комиссия это видит и оценивает негативно
- Не торопиться — лучше чуть медленнее и внятно
- Если сбился — не паниковать, спокойно перейти к следующему пункту
- На вопросы по коду — отвечать конкретно, не уклоняться
